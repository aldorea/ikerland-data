**Section structure (in order):**

1. **Introduction paragraph** — "This section documents how alarm events are detected, deduplicated, and delivered to notification channels. The alarm pipeline begins with IoT Rules Engine threshold evaluation, flows through a Lambda alarm evaluator that prevents notification storms via DynamoDB-based deduplication, fans out through SNS to SES for email delivery, and publishes to EventBridge for future channel extensibility."

2. **Alarm Pipeline Component Inventory** — Table with columns: Component | Service | Responsibility | Invocation. Rows:
   - Threshold Pre-Filter | IoT Rules Engine (AlarmRule) | Coarse threshold filter via SQL WHERE clause — reduces Lambda invocations by filtering non-alarm messages | IoT Core message matching `devices/+/alarm`
   - Alarm Evaluator | AWS Lambda (Python 3.12, arm64) | Per-device threshold check against Aurora rules, deduplication via DynamoDB conditional write, SNS publish, EventBridge event | Async invocation from IoT Rules Engine
   - Dedup Table | Amazon DynamoDB | Stores recent alarm records with TTL for deduplication window | Conditional put_item from alarm evaluator Lambda
   - Notification Hub | Amazon SNS (Standard topic: `alarm-notifications`) | Fans out alarm events to all subscribed notification channels | Published by alarm evaluator after dedup check passes
   - Email Delivery | Lambda → Amazon SES | Renders HTML alarm email template and sends via SES | SNS subscription on `alarm-notifications` topic
   - Extensibility Bus | Amazon EventBridge (custom bus: `iot-alarm-bus`) | Future notification channels (SMS, webhook, Slack, PagerDuty) via rules | PutEvents from alarm evaluator Lambda after SNS publish
   - Dead-Letter Queue | Amazon SQS (`sqs-alarm-dlq`) | Captures failed alarm evaluator invocations for retry/debugging | Lambda DeadLetterConfig (async invocation pattern)

3. **IoT Rules Engine Threshold Detection** (per D-08, ALRM-01) subsection:
   - Reference the AlarmRule already documented in `02-device-connectivity-ingestion.md`
   - Show the full SQL: `SELECT *, topic(2) AS thingName, timestamp() AS ruleTimestamp FROM 'devices/+/alarm' WHERE value > 80`
   - Explain: "The WHERE clause acts as a coarse pre-filter. The literal threshold (80) catches obvious breaches before Lambda invocation. The Lambda evaluator then applies per-device thresholds from Aurora alert-rule configurations — enabling different thresholds per device type or location."
   - **Anti-pattern callout:** "Without the WHERE clause, EVERY message on `devices/+/alarm` triggers the Lambda evaluator — including status updates, test messages, and low-value readings. At 1,000 devices, this means 1,000 unnecessary Lambda invocations per hour."

4. **Lambda Alarm Evaluator Flow** (per D-09, ALRM-04) subsection:
   - Step-by-step numbered flow:
     1. Receive alarm event from IoT Rules Engine (async invocation)
     2. Extract `thingName` and `alarmType` from event payload
     3. (Optional) Look up per-device threshold from Aurora `alert_rules` table via RDS Proxy
     4. If value does not exceed per-device threshold, discard (log and return)
     5. Attempt DynamoDB conditional write: `put_item` with `ConditionExpression = "attribute_not_exists(PK)"` on key `DEDUP#{thingName}#{alarmType}`, TTL = current_time + 900 (15 minutes)
     6. If conditional write succeeds (new alarm): publish to SNS `alarm-notifications` topic AND call EventBridge `PutEvents` on `iot-alarm-bus`
     7. If conditional write fails (ConditionalCheckFailedException — duplicate within 15-min window): skip SNS/EventBridge, increment CloudWatch metric `AlarmsDeduplicated`
     8. On any unhandled error: Lambda async retry (2 retries) then routes to SQS `sqs-alarm-dlq`

5. **Deduplication Pseudo-Code** (per D-09) — Python pseudo-code block copied from 02-RESEARCH.md:
```python
try:
    table.put_item(
        Item={
            "PK": f"DEDUP#{thing_name}#{alarm_type}",
            "TTL": int(time.time()) + 900,  # 15 minutes
            "firstAlarmTs": event_ts,
            "alarmValue": value
        },
        ConditionExpression="attribute_not_exists(PK)"
    )
    # Condition succeeded -> first alarm in window -> publish to SNS
    sns.publish(TopicArn=ALARM_TOPIC_ARN, Message=json.dumps(alarm_payload))
except ConditionalCheckFailedException:
    # Duplicate alarm within 15-minute window -> suppress
    cloudwatch.put_metric_data(Namespace="IoT/Alarms", MetricData=[
        {"MetricName": "AlarmsDeduplicated", "Value": 1, "Unit": "Count"}
    ])
```

6. **SNS Fan-Out and SES Email** (per D-10, ALRM-02) subsection:
   - SNS Standard topic `alarm-notifications` — $0.50/million publishes
   - Current subscription: Lambda `ses-email-sender` → SES `SendTemplatedEmail`
   - Why SNS over direct Lambda→SES: "SNS decouples the alarm evaluator from notification delivery. Adding a new channel requires only a new SNS subscription or EventBridge rule — no changes to alarm evaluator code."
   - SES details: HTML email templates, bounce/complaint tracking, domain verification required, $0.10/1K emails

7. **EventBridge Extensibility** (per D-11, ALRM-03) subsection:
   - Custom event bus: `iot-alarm-bus`
   - Alarm evaluator calls `eventbridge:PutEvents` after SNS publish (both fire for every alarm)
   - EventBridge PutEvents example copied from 02-RESEARCH.md:
```python
eventbridge.put_events(
    Entries=[{
        "Source": "iot.alarm-evaluator",
        "DetailType": "AlarmTriggered",
        "Detail": json.dumps({
            "thingName": thing_name,
            "alarmType": alarm_type,
            "value": value,
            "threshold": threshold,
            "ts": event_ts
        }),
        "EventBusName": "iot-alarm-bus"
    }]
)
```
   - Future channel targets (document as extensibility points, not detailed implementations per D-11):
     - EventBridge rule → Lambda → SNS SMS (for on-call alerts)
     - EventBridge rule → API Destination (webhook to Slack, PagerDuty)
     - EventBridge Pipes → third-party integrations
   - Note per D-12: "Rate-of-change detection and multi-signal correlation are v2 enhancements achievable via EventBridge pattern matching rules — not included in the initial architecture."

8. **Alarm Pipeline Mermaid Diagram** (per D-15) — `flowchart LR` showing:
   - Device --> IoT Core --> AlarmRule[IoT Rules Engine<br/>WHERE value > 80]
   - AlarmRule --> Lambda[alarm-evaluator Lambda]
   - Lambda --> DynamoDB{DynamoDB<br/>dedup check}
   - DynamoDB -->|new alarm| SNS[SNS alarm-notifications]
   - DynamoDB -->|duplicate| CW[CloudWatch<br/>AlarmsDeduplicated]
   - SNS --> SES[Lambda → SES<br/>email notification]
   - Lambda --> EB[EventBridge<br/>iot-alarm-bus]
   - EB -.-> Future[Future channels<br/>SMS, webhook, Slack]
   - Lambda -.-> DLQ[SQS sqs-alarm-dlq]
   - Use dashed lines for failure/future paths

9. **DLQ Configuration for Alarm Evaluator** — Reference the DLQ table from 04-data-pipeline-processing.md but reiterate for the alarm evaluator specifically:
   - Lambda function-level `DeadLetterConfig` (NOT ESM — alarm evaluator is invoked asynchronously by IoT Rules Engine, not from Kinesis ESM)
   - SQS `sqs-alarm-dlq`, maxReceiveCount implicit via Lambda retry policy (2 retries for async, then DLQ)
   - CloudWatch alarm: `ApproximateNumberOfMessagesVisible > 0` triggers ops notification

10. **Design Notes** — Bullet list:
    - The alarm pipeline has TWO deduplication gates: (1) IoT Rules Engine WHERE clause (coarse), (2) DynamoDB conditional write (per-device, per-alarm-type, 15-min window)
    - EventBridge is the extensibility hook — adding SMS/webhook/Slack requires only an EventBridge rule, no code changes to the alarm evaluator
    - DynamoDB TTL provides free cleanup of dedup records — no scheduled job needed
    - The alarm evaluator Lambda has a separate IAM role from the telemetry processor — least privilege per function
    - CloudWatch metric `AlarmsDeduplicated` enables monitoring of dedup effectiveness — a sustained high rate may indicate a device firmware bug or misconfigured threshold
