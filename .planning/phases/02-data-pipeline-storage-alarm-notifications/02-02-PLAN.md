**Section structure (in order):**

1. **Introduction paragraph** — "This section documents the three-tier storage architecture that serves the IoT platform's hot-path data needs. Each tier is purpose-built for its access pattern: Timestream for time-series telemetry queries, DynamoDB for low-latency key-value lookups, and Aurora Serverless v2 for relational data. All storage services are deployed in VPC private data subnets with no public endpoints — access is restricted to Lambda functions and internal services running within the VPC."

2. **Storage Tier Overview Table** — Table with columns: Tier | Service | Data Stored | Access Pattern | VPC Access. Three rows:
   - Hot Time-Series | Amazon Timestream for LiveAnalytics | Device telemetry (temperature, humidity, etc.) | Time-range queries, aggregations, dashboard panels | Interface VPC endpoint (~$7/month per AZ)
   - Operational State | Amazon DynamoDB (On-Demand) | Device metadata, latest-value cache, command queue, alarm dedup records, config state | Single-digit ms key-value lookups, conditional writes | Gateway VPC endpoint (free)
   - Relational | Amazon Aurora PostgreSQL Serverless v2 | Users, roles, alert rule configurations, device groups | SQL queries with joins, complex conditions | Private data subnet, no public endpoint

3. **Amazon Timestream for LiveAnalytics** (per D-05, STOR-01) subsection:
   - Memory store: configurable retention (recommend 24 hours — covers most dashboard lookups)
   - Magnetic store: configurable retention (recommend 365 days for historical analysis)
   - **Critical note:** "Data is tiered from memory to magnetic automatically — not deleted. Memory tier queries have no per-query cost; magnetic tier queries cost $0.01/GB scanned."
   - Write pricing: $0.50 per million records. Batch all records from Lambda invocation into a single WriteRecords API call.
   - VPC access: Interface VPC endpoint required. Timestream has no Gateway endpoint option.
   - Table structure: Database `iot-telemetry`, Table `device-metrics`. Dimensions: thingName, deviceType, location. Measures: temperature, humidity, etc.

4. **Amazon DynamoDB** (per D-05, STOR-02) subsection:
   - Five table roles documented in a table with columns: Table/Usage | Partition Key | Sort Key | TTL | Purpose:
     - device-metadata | `DEVICE#{thingName}` | `META` | No | Device type, location, firmware version, registration date
     - device-latest | `DEVICE#{thingName}` | `LATEST` | 1 hour | Most recent telemetry values for dashboard instant lookups
     - command-queue | `DEVICE#{thingName}` | `CMD#{timestamp}` | 24 hours | Pending commands awaiting device reconnection (complement to Device Shadow)
     - alarm-dedup | `DEDUP#{thingName}#{alarmType}` | None (PK only) | 15 minutes (900s) | Deduplication window for alarm notifications
     - config-state | `DEVICE#{thingName}` | `CONFIG` | No | Last applied configuration version and acknowledgment status
   - On-Demand billing: no capacity planning, auto-scales with traffic. $1.25/million write RCU, $0.25/million read RCU.
   - TTL: Free deletion of expired items. Used for dedup records (15 min), command queue (24h), latest-value cache (1h).
   - VPC access: Gateway VPC endpoint (free, no per-hour charge).

5. **Amazon Aurora PostgreSQL Serverless v2** (per D-05, STOR-03) subsection:
   - Use case: Relational data requiring joins and complex queries — users, roles (RBAC), alert rule configurations (per-device thresholds, notification preferences), device groups.
   - Scaling: 0.5 ACU minimum to 8 ACU maximum (configurable). **"Aurora Serverless v2 scales to a minimum of 0.5 ACU when idle — not zero. Monthly minimum cost: ~$0.06/hour x 720 hours = ~$43/month."**
   - PostgreSQL dialect: Standard SQL, compatible with existing tools and ORMs.
   - RDS Proxy: **Required** for Lambda connection pooling. "Lambda cold starts create new TCP connections to Aurora. Without RDS Proxy, concurrent Lambda invocations can exhaust Aurora's connection limit (~300 for small ACU configurations). RDS Proxy is required in production."
   - Credentials: AWS Secrets Manager for auto-rotation. Lambda retrieves credentials via SDK at startup — no hardcoded credentials.
   - VPC access: Deployed in private data subnets. No public endpoint. Security group allows inbound PostgreSQL (port 5432) only from Lambda security group.

6. **VPC Placement Diagram** — Mermaid `flowchart TD` diagram showing:
   - Subgraph "VPC" containing:
     - Subgraph "Private Subnets (compute)" with Lambda telemetry-processor, Lambda alarm-evaluator
     - Subgraph "Data Subnets (storage)" with Aurora PostgreSQL + RDS Proxy, DynamoDB (via Gateway endpoint), Timestream (via Interface endpoint)
   - Arrows from Lambda functions to each storage service through VPC endpoints
   - Reference: "See 01-security-foundation.md for complete VPC topology, subnet CIDRs, and route tables"

7. **Time-Series Storage Comparison Table** (per D-07, STOR-04) — Table with columns: Alternative | Pros | Cons | Cost Profile | Recommendation. Four rows:
   - **Amazon Timestream for LiveAnalytics** | Purpose-built for time-series; 1,000x faster range queries than RDS; automatic memory-to-magnetic tiering; serverless | Vendor lock-in; limited SQL dialect; no ACID transactions | Writes: $0.50/million; Memory queries: free; Magnetic queries: $0.01/GB; Magnetic storage: $0.03/GB-month | **Recommended** — best fit for IoT telemetry range queries and aggregations at scale
   - DynamoDB (time-sorted SK) | Sub-millisecond point lookups; flexible schema; Gateway endpoint (free) | Range queries require careful schema design (PK=deviceId, SK=timestamp); no native aggregation functions; more expensive for large scans | On-Demand: $1.25/million write RCU, $0.25/million read RCU | Use if already heavily invested in DynamoDB and need single-digit ms point lookups
   - InfluxDB on EC2 | Rich Flux query language; open-source community; purpose-built for time-series | Requires EC2 instance management; backup/HA is manual; no native AWS integration; not serverless | EC2 instance cost ($50-200/month) + EBS storage | Avoid for AWS-first design — adds operational overhead without native integration
   - RDS PostgreSQL (time-series extension) | Standard SQL; pg_timescaledb extension available; familiar tooling | Not purpose-built; slower time-range queries at scale; manual partitioning; connection limit pressure from Lambda | RDS instance cost ($30-200/month) or Aurora ACU pricing | Use only if PostgreSQL is a hard organizational requirement

8. **Relational Storage Comparison Table** (per D-07, STOR-04) — Table with columns: Alternative | Pros | Cons | Cost Profile | Recommendation. Four rows:
   - **Aurora PostgreSQL Serverless v2** | Scales with demand (0.5-8 ACU); PostgreSQL standard; managed HA with Aurora storage; compatible with RDS Proxy for Lambda | 0.5 ACU minimum (not zero-cost at idle); ~$43/month minimum; Serverless v2 is newer (fewer community resources) | $0.12/ACU-hour; ~$43/month minimum at idle | **Recommended** — best balance of cost, scalability, and standard SQL for relational IoT data
   - RDS PostgreSQL Provisioned | Lower per-hour cost at steady load; well-documented; Reserved Instances available | Cannot scale down at idle; minimum db.t3.micro ~$13/month but limited performance; manual scaling | $13-200/month depending on instance | Use for production workloads with predictable 24/7 load
   - DynamoDB (fully NoSQL) | True zero-cost at zero traffic; sub-ms lookups; infinite scaling | No native joins; aggressive denormalization required for relational data (users + roles + alert rules); complex queries become application logic | On-Demand: pay per request only | Use only if fully NoSQL architecture is a hard requirement and all data can be denormalized
   - Aurora PostgreSQL Provisioned | Highest performance (Aurora storage engine); Multi-AZ HA included; Reader replicas | Cannot scale down to zero or near-zero; minimum ~$60/month; over-provisioned for a design exercise workload | $0.08-0.32/ACU-hour provisioned | Use for high-throughput production with predictable capacity needs

9. **Design Notes** — Bullet list:
   - Three-tier separation ensures each storage service handles its optimal access pattern — no single-service compromise
   - DynamoDB TTL provides free cleanup of transient data (dedup, command queue) without scheduled jobs
   - RDS Proxy is not optional for Lambda → Aurora — connection exhaustion is the #1 production issue with this pattern
   - All VPC endpoints are documented in the Security Foundation (01-security-foundation.md) — this section references that placement
   - Timestream memory store retention should be tuned based on dashboard query patterns — 24 hours covers typical operational views
