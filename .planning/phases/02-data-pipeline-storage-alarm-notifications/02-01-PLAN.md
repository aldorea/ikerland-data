6. **SQS Dead-Letter Queue Configuration** (per D-04, PROC-02) — Table with columns: Consumer Lambda | DLQ Name | maxReceiveCount | Retention | CloudWatch Alarm. Two rows:

   - alarm-evaluator | sqs-alarm-dlq | 3 | 14 days | `ApproximateNumberOfMessagesVisible > 0` triggers SNS alert to ops
   - Add an **important note** below the table: "For Kinesis ESM consumers (telemetry-processor), the DLQ is configured via `DestinationConfig.OnFailure` on the Event Source Mapping resource, NOT on the Lambda function configuration. Lambda function-level `DeadLetterConfig` applies only to asynchronous invocations. For the alarm-evaluator (invoked asynchronously by IoT Rules Engine), `DeadLetterConfig` IS configured on the Lambda function. Both DLQ types must be present — this distinction is critical."

7. **Design Notes** — Bullet list:
   - Lambda processes both telemetry and alarm messages via different invocation paths (Kinesis ESM for telemetry, IoT Rules Engine async for alarms)
   - Timestream WriteRecords called once per batch (not per record) to minimize write unit cost
   - DynamoDB latest-value cache enables instant dashboard lookups without Timestream range queries
   - arm64 (Graviton2) runtime selected for 20% cost reduction at equivalent performance
   - Cold start amortization: at batch size 200, one cold start costs 1/200th of per-message invocation overhead

**Important anti-patterns to document (add as "Avoid" callout box):**

- Do NOT send one Timestream WriteRecords call per record — batch all records from the Kinesis batch
- Do NOT use Kinesis Data Analytics (deprecated) — use Lambda for transformations
