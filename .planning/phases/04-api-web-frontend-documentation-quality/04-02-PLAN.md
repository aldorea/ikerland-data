**1. Section intro (## Web Frontend):** Describe the SPA's role — the browser-based dashboard that IoT platform operators use to monitor device telemetry, manage devices, view/acknowledge alarms, and send commands to devices. The SPA is a static JavaScript application (React/Vue/Angular — framework choice is implementation detail, not architecture) hosted on S3, served through CloudFront with WAF protection. All dynamic data comes from the REST API documented in 08-api-layer.md. Per D-06.

**2. Hosting Architecture (### SPA Hosting Architecture):** Document:
- S3 private bucket (Block All Public Access enabled) stores the built SPA assets (HTML, JS, CSS, images).
- CloudFront distribution serves as the only access path to S3 via Origin Access Control (OAC). Per D-07.
- ACM certificate for HTTPS on custom domain (e.g., `dashboard.iot-platform.example.com`). Certificate in us-east-1 for CloudFront. Per D-05 from API layer, same pattern.
- Route 53 alias record pointing custom domain to CloudFront distribution.
- WAF WebACL attached to CloudFront distribution — same rules as documented in 01-security-foundation.md (SEC-06): rate limiting, SQL injection, XSS, geo-blocking. Cross-reference only, do not re-document WAF rules. Per D-07.

**3. CloudFront OAC Configuration (### Origin Access Control):** Per D-07 and WEB-03, document:
- OAC replaces the legacy OAI (Origin Access Identity). OAC is the current AWS recommendation (GA August 2022).
- OAC uses the `cloudfront.amazonaws.com` service principal with an `AWS:SourceArn` condition scoped to the specific distribution.
- Include the S3 bucket policy snippet VERBATIM from RESEARCH.md:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontOAC",
      "Effect": "Allow",
      "Principal": { "Service": "cloudfront.amazonaws.com" },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::iot-dashboard-spa/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::123456789012:distribution/EXAMPLEID"
        }
      }
    }
  ]
}
```

- Add a note: "Replace `iot-dashboard-spa`, `123456789012`, and `EXAMPLEID` with actual values. The key difference from legacy OAI: OAC uses the `Service` principal instead of a `CanonicalUser` principal."

**4. Cache Behavior Table (### CloudFront Cache Behaviors):**

| Path Pattern | Origin | Cache TTL | Notes |
|-------------|--------|-----------|-------|
| `/index.html` | S3 (OAC) | Short (60s) | SPA entry point — must reflect deployments quickly |
| `/static/*` | S3 (OAC) | Long (1 year) | Hashed filenames (e.g., `main.a1b2c3.js`) — immutable |
| `/api/*` | API Gateway HTTP API | No cache (forward all) | Dynamic data — Authorization header forwarded |
| Default (`*`) | S3 (OAC) | Medium (1 hour) | Other assets (images, fonts) |

**6. Web-to-Device Command Flow Sequence Diagram (### Web-to-Device Command Flow):** Per D-08 and WEB-02, create a Mermaid `sequenceDiagram` showing the COMPLETE end-to-end command flow. This ties Phase 1 (Device Shadow) and Phase 4 (API) together:

```
participant Browser as Operator Browser
participant APIGW as API Gateway HTTP API
participant Lambda as Lambda (command-handler)
participant Shadow as IoT Device Shadow
participant DDB as DynamoDB (audit log)
participant Device as IoT Device

Note over Browser,APIGW: User already authenticated via Cognito (see Authentication Flow in 08-api-layer.md)

Browser->>APIGW: POST /devices/{thingName}/commands {action: "reboot"}
APIGW->>APIGW: JWT authorizer validates token
APIGW->>Lambda: Invoke in VPC private subnet
Lambda->>Shadow: UpdateThingShadow (desired: {command: "reboot"})
Shadow-->>Lambda: 200 OK (version N)
Lambda->>DDB: PutItem (command audit: thingName, timestamp, action, user)
Lambda-->>APIGW: 202 Accepted {message: "Command queued", shadowVersion: N}
APIGW-->>Browser: HTTP 202 JSON

Note over Shadow,Device: Device reconnects hourly (see Device Shadow sequence in 03-device-management.md)

Device->>Shadow: GET /things/{thingName}/shadow
Shadow-->>Device: Delta: {command: "reboot"}
Device->>Device: Execute command
Device->>Shadow: ReportThingShadow (reported: {command: "reboot", status: "completed"})
Shadow-->>Shadow: Delta cleared (desired == reported)
```

Below the diagram, add: "The first half (Browser → Shadow update) is the API-layer contribution documented here. The second half (Device reconnects → delta apply → reported update) is the Device Shadow pattern documented in 03-device-management.md. Together, they form the complete web-to-device command delivery lifecycle required by DEVM-02."

**7. Comparison Table: Web Hosting (### Web Hosting Comparison):** Per D-06:

| Criterion | SPA on S3 + CloudFront | Amazon Managed Grafana | AWS Amplify Hosting |
|-----------|------------------------|----------------------|---------------------|
| Use Case | Custom IoT dashboard with device command UI | Operations-only monitoring (time-series visualization) | Developer-friendly SPA deployment |
| Customization | Full control (custom components, command flows, branding) | Limited to Grafana plugins and panels | Full control (wraps S3 + CloudFront) |
| Device Commands | Yes — custom UI → API → Device Shadow | No — Grafana is read-only visualization | Yes — same as S3 + CloudFront |
| Data Sources | Any (via REST API) | Direct Timestream, CloudWatch, DynamoDB plugins | Any (via REST API) |
| Authentication | Cognito User Pools (JWT) | AWS IAM / SAML / Cognito Identity Pools | Cognito User Pools |
| Cost | S3 + CloudFront (~$1-5/month) | $9/editor/month + $5/viewer/month | S3 + CloudFront + CI/CD (~$1-5/month + build minutes) |
| CI/CD | Manual or custom pipeline | Not applicable (managed) | Built-in (branch previews, auto-deploy) |
| Infrastructure Visibility | Explicit (shows S3, CloudFront, OAC) — ideal for architecture assessment | Abstracted (managed service) | Abstracted (wraps S3 + CF) |
| **Recommendation** | **Selected** — custom dashboards, device commands, architecture transparency for assessment | Best for ops-only teams needing quick Timestream dashboards without custom UI | Consider for production deployments where CI/CD convenience > architecture transparency |

- ACM wildcard certificate: `*.iot-platform.example.com` covers both `dashboard.iot-platform.example.com` (CloudFront) and `api.iot-platform.example.com` (API Gateway).
- Cross-references: "See 01-security-foundation.md for WAF rules and VPC topology. See 03-device-management.md for the full Device Shadow delta sequence. See 08-api-layer.md for API endpoints and Cognito authentication flow."
