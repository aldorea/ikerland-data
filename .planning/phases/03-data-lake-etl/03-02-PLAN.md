**Section 7 — Lake Formation Access Control (per D-15)**

Narrative explaining Lake Formation's role: governs multi-team access to the Data Lake beyond raw S3 bucket policies. Provides column-level security (restrict which columns a team can query) and row-level filters (restrict which partitions a team can access).

Include a brief example: "A maintenance team can query `device_health` Gold data but cannot access raw telemetry in Silver. An analytics team can query all Silver columns except `thing_name` (PII masking)."

Note that Lake Formation intercepts Athena and Glue queries, enforcing access before the query reaches S3. This is more granular than S3 bucket policies, which can only control access at the object/prefix level.

Per D-15: Recommended as the access control layer over raw S3 bucket policies for production multi-team environments.

**Section 8 — Design Notes and Anti-Patterns**

Include these anti-patterns as a bulleted list or table (sourced from 03-RESEARCH.md anti-patterns section):

1. **Never query Bronze via Athena**: Bronze JSON has no columnar optimization. Every query scans every byte. A Bronze query costs 8-10x more than the equivalent Silver Parquet query. Use `BytesScannedCutoffPerQuery` as a safety net.

2. **Never partition by device_id**: Thousands of device-specific partitions create the small-file problem. Each partition produces tiny files. Glue and Athena pay per S3 LIST call, and small files degrade query performance. Partition by `device_type` (3-10 values) not `device_id` (thousands).

3. **Never write ETL output to Bronze**: Bronze is the immutable audit trail and replay source. Enforce via S3 bucket policy: deny `s3:PutObject` on `bronze/` prefix for the Glue ETL IAM role. The ETL role should have `s3:GetObject` on Bronze, `s3:PutObject` on Silver only.

4. **Never rely solely on Glue Crawler for schema management**: Crawlers can introduce schema drift when new fields appear. For a known, stable IoT telemetry schema, define the Glue Catalog table schema explicitly. Use Crawlers only for initial schema discovery.

5. **Always configure Glue job failure alerting**: Glue jobs that fail silently leave Silver/Gold stale. Configure CloudWatch Alarm on the Glue job `Failed` metric. Route alarm to SNS for operator notification.

6. **Always create Glue Catalog table before Firehose Parquet conversion**: Firehose Parquet conversion requires a pre-existing Glue Catalog table. Without it, Firehose silently falls back to JSON delivery. Document this as a setup ordering requirement.
